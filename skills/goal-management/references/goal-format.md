## 目標記載フォーマット

**重要**: 各目標には必ず対応する評価項目を明記すること

## 「目標」の定義

目標は「具体的な成果」を記載する。手段やツールの使用を目標にしない。

### 悪い例（手段を目標にしている）
- ❌ 「〇〇を仕組み化する」
- ❌ 「macloggerで記録する」
- ❌ 「learning-supportを使う」
- ❌ 「〇〇のスキルを作る」（目的が不明確）

### 良い例（具体的な成果を目標にしている）
- ✅ 「Rubyの基礎・発展学習を完了する」（learning-supportは手段として記載）
- ✅ 「技術書2冊を読破する」
- ✅ 「個人開発プロジェクトを完成させる」
- ✅ 「article-generatorスキルを作成する」（記事執筆の効率化という目的が明確）

**原則**: ツールの使用は目標ではなく、目標達成の手段として括弧書きで記載する。

## 月次目標のID付与ルール

月次目標には必ずIDを付与し、週次目標との対応を明確にする。

**IDフォーマット**: `[月-value-番号]`
- 月: 1月、2月、3月、4月、5月、6月など
- value: hungry、owner、sustainable
- 番号: 同じvalueカテゴリ内での連番

### 月次目標のフォーマット例

```markdown
## 2026年1月の目標

### Be hungry（貪欲になれ）

#### 【評価項目1: 貪欲に新しいものを勉強したか】
- [ ] **[1月-hungry-1] Rubyの基礎・発展学習を完了する**（learning-support使用）
  - 今週中にGitHub Issuesを全て作成
  - 各Issueを7ステップサイクルで完了させる
- [ ] **[1月-hungry-2] 技術書2冊を読破する**
  - 「良いコード／悪いコードで学ぶ設計入門」を読み終える
  - 「達人に学ぶDB設計徹底指南書」を読み終える

#### 【評価項目2: 貪欲に仕事に取り組んだか】
- [ ] **[1月-hungry-3] 個人開発プロジェクトでRuby(Rails)の成果物を作成する**
  - プロジェクトの要件定義・仕様作成
  - 基本機能の実装（spec-to-codeでSDD+TDD実践）

### Be an owner（当事者たれ）

#### 【評価項目1: 締め切り意識を持てたか】
- [ ] **[1月-owner-1] 個人開発プロジェクトのタスクを期限内に完了させる**
  - task-managementでGitHub Projectsを作成
  - タスクを適切な粒度に分解し、優先順位を設定

#### 【評価項目2: チーム全体の生産性に寄与したか】
- [ ] **[1月-owner-2] article-generatorスキルを作成する**
  - macloggerの記録から記事ネタを提案する機能
  - 段落構成を作成する機能
  - 2月の記事執筆で使用できる状態にする

### Be sustainable（継続的であれ）

#### 【評価項目1: 属人化しないように仕組み化するようになにかやれたか】
- [ ] **[1月-sustainable-1] article-generatorで記事執筆プロセスを仕組み化する**
  - 誰でも「何を書けばいいか」がわかる仕組みを作る
  - 組織貢献の視点を自動で提案する機能を実装

#### 【評価項目2: 日常業務に対して改善する動きができたか】
- [ ] **[1月-sustainable-2] learning-supportで学習の効率化を実践する**
  - Ruby学習をlearning-supportで完了させる
  - 学習時間・効果を記録し、効率化の効果を測定
```

## 週次目標と月次目標の対応

週次目標には、対応する月次目標のIDを明記する。

### 週次目標のフォーマット例

```markdown
### 1月第1週（1/5-1/11）

#### Be hungry（貪欲になれ）
- [ ] [1月-hungry-1] Ruby学習Day 1を完了させる（learning-supportで7ステップサイクル）
- [ ] [1月-hungry-2] 「良いコード／悪いコードで学ぶ設計入門」を読み進める（3-5章を読了）
- [ ] [1月-hungry-2] 「達人に学ぶDB設計徹底指南書」を読み始める（1-2章を読了）

#### Be an owner（当事者たれ）
- [ ] [1月-hungry-3] 個人開発プロジェクトの要件を整理する（何を作るか決定）

#### Be sustainable（継続的であれ）
- [ ] （今週は特になし）
```

**重要**: 週次目標のタスク冒頭に月次目標IDを付けることで、どの月次目標に対応しているか一目で分かる。

## 半期目標のフォーマット例

半期目標にはIDは不要だが、評価項目との対応は必須。

```markdown
## Be hungry（貪欲になれ）

### 【評価項目1: 貪欲に新しいものを勉強したか】

- [ ] 設計関連の技術書を2冊以上読破（DB設計、アーキテクチャパターンなど）
- [ ] 学んだ設計パターンを実プロジェクトで3回以上適用
- [ ] AWSの体系的な学習を行う（AWS認定資格取得を目指す）

### 【評価項目2: 貪欲に仕事に取り組んだか】

- [ ] 技術ブログ/Qiita記事を隔週で投稿 → **半期で12本以上**
- [ ] 詳細な技術ドキュメントを2本作成（Confluence/Google Slide/動画解説など）
- [ ] AIを使った設計の最適化手法をドキュメント化

## Be an owner（当事者たれ）

### 【評価項目1: 締め切り意識を持てたか】

- [ ] すべてのタスクを期限内に完了（やむを得ない場合は事前相談）
- [ ] 週次報告を欠かさず実施

### 【評価項目2: チーム全体の生産性に寄与したか】

- [ ] 月1回のチーム勉強会を主催
- [ ] コードレビューを積極的に実施（週5件以上）
- [ ] 新メンバーのオンボーディング支援

## Be sustainable（継続的であれ）

### 【評価項目1: 属人化しないように仕組み化するようになにかやれたか】

- [ ] 開発手順書を5本以上作成
- [ ] 既存の属人化タスクを2つ以上ドキュメント化・引き継ぎ

### 【評価項目2: 日常業務に対して改善する動きができたか】

- [ ] CI/CDパイプラインの改善（ビルド時間20%短縮）
- [ ] 繰り返し作業の自動化スクリプトを3つ以上作成
```
