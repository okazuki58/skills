# タスク管理 - 例

## 例1: 既存プロジェクトのタスク整理

```
User: "このリポジトリのタスク管理をしたい"

Claude:
[リポジトリ分析]
"リポジトリを分析しました。

📊 分析結果:
- プロジェクト: Rails ECサイト
- 技術スタック: Ruby 3.2, Rails 7.1, PostgreSQL, React
- 既存のIssue: 15件（オープン: 8、クローズ: 7）
- 最近のコミット: アクティブな開発（週5-10コミット）

タスク管理を開始しましょう。"

[対話的計画立案]
Q1: "目標は？" → User: "新機能、既存機能の改善"
Q2: "優先度は？" → User: "新機能"
Q3: "タイムラインは？" → User: "1ヶ月以内"
Q4: "チームサイズは？" → User: "2-3人"
Q5: "粒度は？" → User: "中（タスクあたり2-3日）"

[タスク分解]
"分解を作成しました:

### 大タスク1: ユーザー認証
- 中1-1: 認証ロジック
  - [ ] #16: Deviseをインストール
  - [ ] #17: ユーザー登録画面
  - [ ] #18: ログイン画面
- 中1-2: 認証後の処理
  - [ ] #19: セッション管理
  - [ ] #20: 認証必須ページの保護

### 大タスク2: 商品検索改善
- 中2-1: 検索ロジックの最適化
  - [ ] #21: Elasticsearchの評価
  - [ ] #22: 検索インデックスの作成
- 中2-2: 検索UIの改善
  - [ ] #23: フィルター機能の追加
  - [ ] #24: ページネーションの追加

この計画で進めますか？" → User: "はい"

[GitHub Projects + Issues 作成完了]
"タスク管理の準備ができました！

📊 Projects: https://github.com/{owner}/{repo}/projects/1
📝 Issues: 9件作成

次のタスク:
- Issue #16: Deviseをインストール（優先度: 高）
- Issue #21: Elasticsearchを評価（優先度: 中）

どちらから始めますか？"
```

## タスク分解構造の例

「ユーザー認証機能を追加」の場合:

```
大タスク: ユーザー認証機能
├── 中タスク1: バックエンド認証
│   ├── 小 #1: Devise gemをインストール
│   ├── 小 #2: Userモデルを生成
│   ├── 小 #3: Deviseを設定
│   └── 小 #4: 認証テストを追加
├── 中タスク2: フロントエンド画面
│   ├── 小 #5: 登録フォーム
│   ├── 小 #6: ログインフォーム
│   ├── 小 #7: パスワードリセットフロー
│   └── 小 #8: ユーザープロフィールページ
└── 中タスク3: 認可
    ├── 小 #9: ユーザーロールを定義
    ├── 小 #10: 権限を実装
    └── 小 #11: 認可テストを追加
```

## Issue説明テンプレート

```markdown
## 概要
Deviseを使用したユーザー登録機能の実装

## タスク
- [ ] Devise gemをインストール
- [ ] DeviseでUserモデルを生成
- [ ] Devise initializerを設定
- [ ] 登録ルートを作成
- [ ] 登録フローをテスト

## 受け入れ基準
- [ ] ユーザーがメール/パスワードで登録できる
- [ ] メール検証が機能する
- [ ] パスワード確認が機能する
- [ ] すべてのテストが通る

## 依存関係
- #1 Railsプロジェクトセットアップ（先に完了する必要あり）

## 参照
- [Deviseドキュメント](https://github.com/heartcombo/devise)
```
